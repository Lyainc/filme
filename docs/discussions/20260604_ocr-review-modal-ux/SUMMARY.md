## 합의 사항

### TOPIC 1 — 좌표 기반 박스 탭의 기술 타당성
- 좌표계 일치·오버레이·히트테스트는 **표준 구현, 타당**. 단 현재 전처리 이미지를 버리는 구조라 **보관 로직 추가** 필요.
- **박스 정확도는 미검증 리스크.** `gpt-4o-mini`의 좌표는 ±5~10% 오차가 흔하고(768px에서 약 38px), 티켓 텍스트 줄 높이(30~40px)를 넘어 **한 줄이 통째로 어긋날 수 있음**. → 가정이 아니라 *측정 대상*.
- **재인식 = 재추첨.** Vision LLM은 비결정적이라 재OCR이 "개선"을 보장하지 않음. UX 문구·동작을 그에 맞게 설계.

### TOPIC 2 — 검수/편집 모달의 실사용성
- **스테이징(OK 전 폼 미반영)은 명백히 옳음 — 유지.** 틀린 자동입력이 기존 폼을 덮어쓰는 사고 방지.
- **모바일 오탭은 물리적 제약** (탭 타겟 권장 44~48px > 티켓 텍스트 박스). 튜닝으로 못 가림 → **비-좌표 폴백 필수**.
- 매핑 조작은 **"박스 탭 → 팝오버로 어느 필드? 선택", 한 번에 하나씩.** N×N 동시 시각화(색깔 선 등)는 인지부하만 키움 — 지양.
- 가치 희석 우려는 **자동매핑 정확도가 높을수록 작아짐**("전부 검수"가 아니라 "예외만 수정"). 제품 맥락(저빈도·고관여 제작 도구)에서 속도보다 경험·안전이 우선.

### TOPIC 3 — 인터랙션 방향 (핵심)
- **하이브리드 채택, 위계 명시**: 칩+드롭다운이 **주(主, 항상 작동)**, 이미지+박스 탭이 **보조(補, 성공 시 가속·실패 시 칩으로 자연 폴백)**.
- 근거: 텍스트 추출·의미 분류는 LLM **강점**, 좌표는 **약점**. 칩은 강점 위에, 박스 탭은 약점 위에 섬 → 매핑 *정확성*은 칩에 둠.
- 박스 오버레이(시각)와 칩 리스트(조작)를 분리하면 **단일 실패점 제거**. 사용자 선호("탭의 즐거움"·"이미지 맥락")는 박스를 보여주고 탭 반응을 주는 것만으로 상당 부분 충족.

### TOPIC 4 — PoC 검증 우선순위
1. **#34 = 박스 정확도 정량 게이트**: 실제 티켓 5~10장(CGV·롯데·메가박스 혼합)으로 "박스 중심점이 올바른 텍스트 위에 있는 적중률" 측정. **기준 ≥70%**. 미달 시 박스 탭은 보조로도 부적격 → 순수 칩으로 후퇴.
2. **칩+드롭다운 검수 UI 병행**: 박스 정확도와 무관(텍스트+분류만 의존). 기반이므로 먼저 세움.
3. 게이트 통과 시 박스 오버레이+탭을 #35에 보조로 얹음. 실패 시 칩만으로 출시·박스 보류. 어느 쪽이든 제품 완성.
