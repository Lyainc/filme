# TOPIC 2 — 검수/편집 모달의 실사용성

쟁점: 자동입력 가치 희석(4), 모바일 터치 정밀도(5), 매핑 인지부하(6)

## Independent Statements
- **[UX Expert]**: 모바일 탭 타겟 권장 44~48px(Apple HIG/Material). 티켓 텍스트 박스는 대부분 그보다 작음 → 오탭 구조적 발생. 히트영역 키우면 인접 박스 겹쳐 모호. 튜닝으로 못 가리는 물리 제약.
- **[Product Strategy Expert]**: 이건 저빈도·고관여 제작 도구(포토티켓 메이커). 가치는 "초고속 입력"이 아니라 "만족스러운 제작 경험". 사용자의 "탭 인터랙션 자체가 좋다"는 이 맥락에서 타당한 제품 직관. 속도로만 재면 안 됨.

## Q&A / 교차
- **[Critical Practitioner]**: 즐거움도 작동해야 즐거움. 오탭 연발이면 짜증으로 전환. 자동입력 본질은 노력 절감인데 검수+수정 끼면 상쇄 — 필드 6~7개 타이핑이 더 빠를 수도.
- **[Optimistic Practitioner]**: OCR 신뢰도에 달림. 자주 틀리면 검수는 필수. 스테이징으로 OK 전까지 폼 안 건드리는 게 덮어쓰기보다 안전.
- **[Moderator→UX]**: 매핑 시각화는?
- **[UX Expert]**: N×N 동시 표시는 과함. "박스 탭 → 팝오버로 어느 필드? 선택", 한 번에 하나가 제일 단순. 색깔 선 연결은 화면만 어지럽힘.
- **[Product Strategy Expert]**: 자동매핑 80% 맞으면 틀린 1~2개만 수정 → "전부 검수"가 아니라 "예외만 수정". 가치 희석 작음.

## Dialectic
- **Thesis**: 검수 모달은 신뢰도를 메우고 스테이징으로 안전성 제공.
- **Antithesis**: 모바일 터치 정밀도·단계 증가가 자동입력 속도 가치를 깎음.
- **Synthesis**: 이 제품 맥락에선 속도보다 경험·안전 우선. 가치 희석은 자동매핑 정확도 높을수록 작아짐. 단 모바일 오탭은 물리 제약 → 비-좌표 폴백 병행 필수.

## 결론
스테이징 명백히 옳음, 유지. 매핑은 "탭→팝오버, 한 번에 하나". 모바일 오탭은 구조적 제약 → 비-좌표 폴백 필수. (전원 합의)
