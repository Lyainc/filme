# TOPIC 1 — 좌표 기반 박스 탭의 기술 타당성

쟁점: 좌표계 일치(1), 박스 정확도(2), 재인식 비결정성(3)

## Briefing
- **[Optimistic Practitioner]**: 좌표계 일치는 표준 문제. 전처리 768px 이미지를 모달에 그대로 띄우고 `displayScale = 표시폭/768`로 박스 스케일. 전처리 이미지만 보관하면 됨. 렌더링은 absolute div 오버레이라 평이.
- **[Critical Practitioner]**: 렌더링이 아니라 박스 데이터 품질이 문제. 좌표가 틀리면 좌표계를 맞춰도 엉뚱한 자리에 박스. 쓰레기를 정밀 배치하는 셈.

## Independent Statements
- **[LLM/Vision Expert]**: Vision LLM은 OCR 엔진이 아님. 정확한 bbox 보장 안 함. 좌표 오차 ±5~10% 흔함 → 768px에서 ~38px, 티켓 줄 높이 30~40px → 한 줄 통째로 어긋날 수 있음. `gpt-4o-mini`는 풀 `4o`보다 grounding 더 약함.
- **[Frontend Engineer]**: 좌표 역변환·오버레이·히트테스트 모두 표준. 우려는 (1) 전처리 이미지를 버리는 현 구조 → 보관 로직 추가, (2) 모바일 핀치줌/팬 유입 시 좌표 변환 4단(원본→전처리→표시→줌)으로 복잡도 급증.

## Q&A
- **[Moderator→Vision]**: 비결정성은 얼마나 심각?
- **[LLM/Vision Expert]**: temperature 0에도 vision 인코딩이 미세하게 흔들려 재OCR 시 좌표·분류 달라짐. "다시 누르면 나아진다" 보장 없음. 오히려 악화 가능, 비용은 매번 발생.
- **[Frontend Engineer]**: 그럼 재인식은 "재추첨". 사용자 멘탈모델과 불일치.

## Dialectic
- **Thesis**: 박스 탭은 구현 가능(렌더링·좌표계 표준).
- **Antithesis**: 그 구현이 신뢰 가능한 박스 데이터에 전적 의존하는데, `gpt-4o-mini` 좌표 정확도가 탭 히트테스트 수준에 못 미칠 공산이 큼.
- **Synthesis**: "만들 수는" 있으나 박스 정확도 검증 전엔 작동 보장 불가. 박스 정확도는 가정이 아니라 측정 대상.

## 결론
좌표계·렌더링 타당. 박스 정확도는 미검증 리스크 → PoC 1차 측정 대상. 재인식은 "재추첨"이므로 UX 문구·동작을 그에 맞게. (전원 합의)
